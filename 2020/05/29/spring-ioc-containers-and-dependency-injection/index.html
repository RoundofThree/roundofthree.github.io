Spring IoC Containers and Dependency Injection

spring-framework
Programming
Publish Date:   2020-05-29
Update Date:   2020-05-29

Read Times:   10 Min

Inversion of Control Containers

Aims:

  • Instantiate and configure objects (can be early instantiation or lazy initialization)
  • Manage dependencies between objects (HAS-A relationships)
metadata + POJO classes -> Application (constructed and wired objects)

The container metadata takes form of XML file, Java annotations or a Java class annotated with @Configuration. The objects are initialized and stored in the container, and these objects are called beans. They can be retrieved from the ApplicationContext with reference to the bean ID and casted to the proper class (BeanFactory is also a container, but has fewer functionalities).

For XML file:

  • Place the Beans.xml or any name in the src folder.
    <?xml version = "1.0" encoding = "UTF-8"?>

``` For Java configuration class: - Define a `AppConfig` class annotated with `@Configuration` and set `@Bean`. 
import org.springframework.context.annotation.*;

@Configuration
public class AppConfig {
  @Bean(initMethod="init", destroyMethod="clean")
  @Scope("prototype")
  public A getA() {
    return new A(getB());
  }

  @Bean
  @Scope("singleton")
  public B getB() {
    return new B();
  }
}

To run the application, call the container in the main method.

org.springframework.context.ApplicationContext // Context Module

ApplicationContext ctx = new AnnotationConfigApplicationContext(HelloWorldConfig.class);
A a = ctx.getBean(A.class);  // get the bean which is of class A

ApplicationContext context = new ClassPathXmlApplicationContext("applicationContext.xml");
A a = context.getBean("A");   // here get the bean with id="A"

Dependency injection vs. Dependency lookup

The Dependency Lookup is an approach where we get the resource after demand. Ways to do it:

  • new a resource
  • Factory method: Executors.newFixedThreadPool(10)
  • From Java Naming Directory Interface (JNDI): using new InitialContext().lookup("java:comp/env").lookup("A"), for example.
    The problems are that:
  • Tight-coupling.
  • Hard to unit test.
    The Dependency Injection is a design pattern that injects the resource via constructor or setter methods. It removes dependency.

Say this is dependency lookup:

public class Course {
  private Teacher teacher;
  public Course() {
    this.teacher = new Teacher();
  }
}

So with dependency injection pattern it is:

public class Course {
  private Teacher teacher;
  public Course(Teacher teacher) {
    this.teacher = teacher;
  }
}

In DI, the dependency bean Teacher is not managed by the Course bean, but it is injected. This makes their classes more independent.

Constructor Injection vs Setter Injection

Setter injection injects the dependencies with setAttribute() methods, where attribute is the name of the attribute. Constructor injection injects the dependency when the object is “newed”. Constructor Injection is used when arguments are compulsory, and setter injection is more flexible. Although setter injection can also mark some fields as required.

In XML, constructor injection fields are tagged <constructor-arg index="0" type="java.lang.String" value="Hello"/>. index and type are optional, and value can be replaced by ref in case of reference to other beans.

Setter injection is tagged with <property/>.

XML based configuration

Bean lifecycle

The bean lifecycle is managed by the container by stating the init-method and destroy-method attributes in <bean/> tag. A default can be set with default-init-method = "init" or default-destroy-method in <beans/> tag. The value of this attribute is the name of the method() owned by the bean.

Bean scopes

Corresponds to attribute scope in <bean/>.

  • "prototype". A new object is created every time context.getBean("beanId") is called.
  • "singleton". The same object is returned for same bean Id. Default.
  • "request", "session", "global-session". The scope of the same object is for a same HTTP request/session/global session.

Inheritance

Via parent attribute in <bean/>. The properties with same name will be inherited. The child value can override parent value.

<?xml version = "1.0" encoding = "UTF-8"?>

<beans xmlns = "http://www.springframework.org/schema/beans"
   xmlns:xsi = "http://www.w3.org/2001/XMLSchema-instance"
   xsi:schemaLocation = "http://www.springframework.org/schema/beans
   http://www.springframework.org/schema/beans/spring-beans-3.0.xsd">

   <!-- Definition for beans -->
   <bean id = "course" class = "com.roundofthree.Course" init-method="init" destroy-method="clean" scope="singleton">
      <property name = "title" value = "Spring Framework Basics"/>
      <property name = "dependency">
         <bean id = "teacher" class = "com.roundofthree.Teacher"/>
      </property>
   </bean>

   <bean id = "banner" class = "com.roundofthree.Banner" parent = "course">  <!-- refer to the parent with beanId -->
     <!-- <property name = "title" value = "Override value"/> -->
   </bean>

</beans>

Refer to other beans in

Two ways:

  • A <bean/> is defined inside of the <property/>.
    <property name = "teacher">
<bean id="teacher" class = "com.roundofthree.Teacher"/>  
</property>
  • The bean is referenced in ref = "beanId" attribute of <property/>.
    <bean id = "teacherid" class = "com.roundofthree.Teacher"/>
<!-- bean definition for "course" bean omitted -->
<property name = "teacher" ref = "teacherid">

Collections: List, Set, Map and Properties

Example:

<property name = "courses">
   <list>
      <value>OS and concurrency</value>
      <value>Programming languages and compilers</value>
      <value>Software development</value>
      <value>Network security</value>
   </list>
</property>
  • List: <list> and <value>
  • Set: <set> and <value>
  • Map: <map> and <entry key="" value=""/>
  • Properties: <props> and <prop key="">Value</prop>

We can also inject bean references with <ref bean="id"/> replacing <value/> (and mix them).

Autowiring

Set autowiring with autowire attribute in <bean/>. It can be:

  • no. Default.
  • byName. The property with name=”name” searches a bean with id=”name”.
  • byType. The property with type=”type” searches for a bean of the class “type”. Exception if more than one found.
  • constructor. Search for beans with same types as constructor arguments.

Factory methods

We can get an object by calling factory methods from class A:

  • Static factory method that returns an object of class A (as in singleton pattern):
    <bean id="A" class="com.roundofthree.A" factory-method="getA"/>
  • Static factory method that returns an object of a class other than A:
    <bean id="B" class="com.roundofthree.A" factory-method="getB"/>
  • Non-static factory method, referring to the bean A equivalent to a.getB():
    <bean id="B" class="com.roundofthree.A" factory-method="getB" factory-bean="A"/>

Java annotation based configuration

Annotation wiring can help XML wiring, but the latter override annotation injection.

First, in the XML:

<?xml version = "1.0" encoding = "UTF-8"?>

<beans xmlns = "http://www.springframework.org/schema/beans"
   xmlns:xsi = "http://www.w3.org/2001/XMLSchema-instance"
   xmlns:context = "http://www.springframework.org/schema/context"
   xsi:schemaLocation = "http://www.springframework.org/schema/beans
   http://www.springframework.org/schema/beans/spring-beans-3.0.xsd
   http://www.springframework.org/schema/context   <!-- schema needed for context module annotations -->
   http://www.springframework.org/schema/context/spring-context-3.0.xsd">

   <context:annotation-config/>
   <!-- bean definitions go here -->

</beans>

Annotations (from org.springframework.beans.factory.annotation)

  • @Required. Add to setter methods to throw BeanInitializationException when this property is not set in configuration.
  • @Autowired. Add to setter or constructor. It autowires byType. It applies @Required directly unless @Autowired(required=false) is specified.
  • @Qualifier("beanId"). Add with @Autowired to match to specific bean to avoid confusion.

Java code configuration

This uses Java annotations but is different from the previous annotations. The previous annotations saves the programmer the need to specify the references to beans, but the beans have to be declared in the XML file. Say A references B, then A and B have to be configured properly with <property/> tags, but the <property/> that refers to bean B is omitted. On the other hand, with Java code configuration, a class is defined as a configuration class and contains methods to return beans.

ApplicationContext ctx = new AnnotationConfigApplicationContext(AppConfig.class);
ctx.register(AdditionalConfig.class);  // add more config classes
ctx.refresh();                         // refresh the context
A a = ctx.getBean(A.class);  // get the bean which is of class A

Annotations (from org.springframework.context.annotation.* )

  • @Configuration to the config class.

  • @Import(AdditionalConfig.class) to the class to import bean definitions, so only one config is supplied to the ApplicationContext.

  • @Bean(initMethod="init", destroyMethod="cleanup") to the method with name equal to method name and return a bean.

  • @Scope("prototype") to method that returns a bean.

    The reference to other beans are hard-coded in the bean method, by calling the “getter” method in the same config class.

Component scanning with stereotype annotations @Component

When using annotation-based configuration, the annotations @Autowire can only resolve dependencies between beans, but beans have to be declared in XML. Using component scanning by annotating classes with @Component("beanId"), the class is associated one-to-one to a bean. First we have to allow scanning to all packages that are annotated:

<beans xmlns = "http://www.springframework.org/schema/beans"
   xmlns:context = "http://www.springframework.org/schema/context"
   xmlns:xsi = "http://www.w3.org/2001/XMLSchema-instance"
   xsi:schemaLocation = "http://www.springframework.org/schema/beans     
   http://www.springframework.org/schema/beans/spring-beans-3.0.xsd
   http://www.springframework.org/schema/context
   http://www.springframework.org/schema/context/spring-context-3.0.xsd">

   <context:component-scan base-package = "com.roundofthree.library.service" />
   <context:component-scan base-package = "com.roundofthree.library.dao" />
   <context:component-scan base-package = "com.roundofthree.library.controller" />
   <!-- <context:annotation-config/> Not needed -->
</beans>

Or in Java-based configuration, mark the config class with @ComponentScan("com.roundofthree.library.service"). The config class could well be empty.

@Repository, @Service and @Controller

They are from Spring MVC. They are annotations that inherit from @Component.

  • @Repository. For DAO concrete classes. Makes the unchecked exceptions eligible to translate to Spring DataAccessException if you declare org.springframework.dao.annotation.PersistenceExceptionTranslationPostProcessor as a bean in application context.
  • @Service. For service-layer classes.
  • @Controller. For controller. Allows @RequestMapping(value="/login", method=RequestMethod.GET) to add to methods. To map URL to instance methods.

Look up the latest Spring Framework documentation.
