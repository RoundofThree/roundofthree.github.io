Linux 1.0⌗

Linux 1.0 source tree:

  • boot: kernel bootstrap code, head.s, setup.s, bootSect.s

  • drivers:

  • fs: file system

  • mm: memory management

  • net: networking

  • include: header files

  • init: the first process executed

  • ipc: System V IPC

  • kernel: system calls

  • lib: library code

  • ibcs: Intel Binary Compatibility Standard. Compatibility library that would allow you to execute binaries built for old SCO systems on Linux. It got removed from Linux on later releases.

  • tools: program that splices three images together into a kernel image that can be booted from PC BIOS startup

  • zBoot

boot⌗

  1. bootsect.S is loaded by bios_start at 0x7c00, copies itself to 0x90000, and jumps there. It then loads setup.S directly after itself (0x90200).
  2. Use BIOS calls to write the message “\nLoading system…\n\n”, reads the system image directly at address 0x10000 from the boot device using BIOS calls, then turns off the floppy drive, saves the cursor position, turns off interrupts, and relocates the system image to address 0.
  3. The addresses of a suitable Interrupt Descriptor Table (IDT) and Global Descriptor Table (GDT) are then loaded into the appropriate 386 registers. These tables are stored as data in the bootstrap code image.
  4. The key-board buffer is emptied. The interrupts are reprogrammed. Then the process is switched to protected mode. It then jumps to execute the relocated system image.
  5. head.S. All of the foregoing code would be overwritten by the page tables. The code then performs a jump to the first address after GDT AND IDT, where a C-style call to _main of init/main.c is performed.

Note: In modern Linux, there is no longer a bootsector in a new bootable partition. Linux needs a bootloader, eg. GRUB, to write to the bootsector. GRUB writes to the MBR (Master Boot Record) of the primary drive. MBR contains the first 512 bytes of a partitioned data storage device; the first 440 bytes correspond to the first stage bootloader code, the remaining bytes correspond to the primary partition table. Each partition where an OS is installed starts with the VBR (Volume Boot Record), which stores the main bootstrapping code. The boot manager, eg. GRUB, lets the user choose the second stage bootloader after the first stage bootloader is executed.

init⌗

start_kernel() performs these tasks. First the setups then enables interrupts.

  1. set_call_gate(&default_ldt,lcall7);
  2. PAGE_ALIGN and page_init
  3. trap_init: system traps
  4. init_IRQ
  5. sched_init: scheduler
  6. kmalloc_init, chr_dev_init, blk_dev_init, sti and calibrate_delay
  7. (Optional) net_dev_init, scsi_dev_init
  8. inode_init, file_table_init
  9. mem_init
  10. buffer_init: block device buffer system
  11. time_init: clock
  12. floppy_init
  13. sock_init
  14. ipc_init: System V IPC
  15. sti
  16. (Optional) Check math emulation
  17. Move from kernel mode to user mode
  18. Fork a child to perform init()
  19. Schedule loop: idle

Note: sti means set interrupt flag.

init() performs these tasks:

  1. setup(): read hard drive partition tables
  2. Open /dev/tty1 in read-write mode
  3. execve /etc/init, /bin/init, /sbin/init
  4. Fork a child to execve /bin/sh with HOME=/usr/root
  5. Wait for the child to die and print its exit code.
  6. sync(): synchronize I/O operations
  7. _exit(0)

Building the kernel⌗

First, configure the kernel.

make ARCH=x86_64 x86_64_defconfig 
make ARCH=x86_64 menuconfig

In our case of a kernel 1.0, simply:

make config

This is because kernel 1.0 does not have an arch directory containing default configs for different architectures.

Second, build the kernel using make and the generated .config file.

make -j8
make -j8 modules_install # install LKMs

In our case,

make -j8
make -j8 dep
make clean
make -j8

Installing the kernel⌗

The kernel image is stored at arch/x86/boot/bzImage for the current release. The kernel image is stored at zImage for kernel 1.0.

To boot the kernel we need a root filesystem. I will experiment with three choices: buildroot, qemu-img + debootstrap, mkfs + bochs.

Buildroot⌗ 

git clone git://git.buildroot.net/buildroot
cd buildroot

make menuconfig # select x86_64 and ext4 for current release
# OR
make qemu_x86_defconfig # select i386 and ext2 for kernel 1.0 
make -j8

The root filesystem is stored at buildroot/output/images/.

qemu-img + debootstrap⌗ 

IMG=qemu-image.img
DIR=mount-point.dir
# create disk image of size 1g
qemu-img create $IMG 1g
# make filesystem EXT2
mkfs.ext2 $IMG
mkdir $DIR
# mount disk image on directory using a loop device
sudo mount -o loop $IMG $DIR
# install a Debian base system into the directory
sudo debootstrap --arch amd64 jessie $DIR
# unmount the directory
sudo umount $DIR
rmdir $DIR

mkfs + bochs⌗

See INFO.TXT.

cd linux-1.0
touch myfs
truncate -s 1474560 myfs
mkfs.ext2 -b 1024 myfs
mkdir myfsdir
sudo mount myfs myfsdir/
sudo mkdir myfsdir/bin/
sudo mkdir myfsdir/dev/
mkdir tmp
cat > tmp/init.c << EOF
#include <linux/unistd.h>
#include <linux/types.h>
int errno;
static inline _syscall3(int,write,int,fd,const char *,buf,off_t,count)
int _start(void){
  write(1, "Hello World!", 12);  
  for(;;);
}
EOF
gcc tmp/init.c -o tmp/init.o -static -c \
	-Wall -Wstrict-prototypes -O2  -fomit-frame-pointer -pipe \
        -w -m32 -I./include/ -std=gnu89 \
        -fno-stack-protector -fno-builtin -mmanual-endbr \
        -fno-reorder-blocks-and-partition -fno-pie
ld -m elf_i386 tmp/init.o -o tmp/init
# place init binary at /bin of rootfs
sudo mv tmp/init myfsdir/bin/
rm -rf tmp
# make a device c 4 1 = /dev/tty1         First virtual console
sudo mknod myfsdir/dev/tty1 c 4 1
sudo umount myfsdir/

Testing the kernel in QEMU⌗

QEMU command arguments:

  • -kernel: kernel image
  • -hda: rootfs image
  • -append: command line arguments supplied to the kernel
  • -s and -S: for remote debugging purposes. Opens port 1234

Without root filesystem:

qemu-system-i386 -no-kvm -kernel zImage -drive file=/dev/zero,format=raw,index=0,media=disk -append "root=/dev/zero console=ttyS0" -serial stdio -display none

With root filesystem and using the rootfs generated by buildroot:

qemu-system-i386 -kernel zImage \
-boot c -m 2049M -drive file=../buildroot/output/images/rootfs.ext2,format=raw,index=0,media=disk \
-append "root=/dev/sda rw console=tty,115200 acpi=off nokaslr" \
-serial stdio -display none

With root filesystem and using the rootfs generated by debootstrap:

qemu-system-x86_64 -kernel zImage \
-hda qemu-image.img -append "root=/dev/sda rw console=ttyS0,115200 acpi=off nokaslr" \
-serial stdio -display none

With root filesystem and using the rootfs generated by mkfs:

qemu-system-i386 -kernel zImage \
-hda myfs -append "root=/dev/sda rw console=ttyS0,115200 acpi=off nokaslr" \
-serial stdio -display none

Same as above but using bochs:
